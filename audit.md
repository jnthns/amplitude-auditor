# Amplitude Taxonomy Audit App — Build Plan

## Architecture Summary

| Layer | Tech | Notes |
|-------|------|-------|
| Framework | React 19 + Vite | TypeScript, single-page app |
| Styling | Tailwind CSS | Warm, vibrant, clean aesthetic |
| LLM | Claude API (client-side) | Sonnet 4 via `anthropic` SDK, API key stored in `.env` / entered at runtime |
| CSV Parsing | PapaParse | Client-side, handles Amplitude's export quirks |
| XLSX Export | SheetJS (xlsx) | Multi-tab output matching Audit Output Template |
| State | Zustand | Lightweight, minimal boilerplate |
| Auth | None | Internal-only, VPN-gated |
| Deploy | Static hosting (Vercel / internal) | No backend needed |

## Page Flow (3 screens)

### Screen 1: Intake Form
**Route:** `/`

Collects everything the audit needs before any LLM calls fire.

**Form fields:**
1. **Customer Name** — text input (required)
2. **Amplitude Project ID** — text input (optional, for report labeling)
3. **Events + Event Properties CSV** — file upload (required, `.csv` only)
4. **User Properties CSV** — file upload (optional, `.csv` only)
5. **Primary Data Source** — multi-select chips (required, one or more):
   - Segment
   - mParticle
   - Rudderstack
   - Amplitude SDK
   - Google Tag Manager
   - CDP / Reverse ETL (Snowflake, BigQuery, Hightouch, Census, etc.)
6. **Industry / Vertical** — dropdown (optional): Fintech, SaaS, E-commerce, Media, Healthcare, Gaming, Other (free text)
7. **Compliance Requirements** — multi-select chips (optional): SOC2, GDPR, HIPAA, PCI-DSS, CCPA, None
8. **Known Concerns** — textarea (optional, free text)
9. **Claude API Key** — password input (required, stored in session only, never persisted)

**On submit:** Parse CSVs client-side → validate schema shape → transition to Screen 2.

### Screen 2: Audit Progress
**Route:** `/audit`

Shows real-time progress as the audit runs across all 45 criteria.

**UI elements:**
- Progress bar with `X / 45 criteria evaluated`
- Criteria area cards (9 areas) that fill in as each area completes
- Each card shows: area name, criteria count, pass/fail tally, running score
- Streaming text display showing the current criterion being evaluated
- Cancel button to abort mid-audit

**LLM execution strategy:**
- Batch criteria by area (9 batches of 3–7 criteria each)
- Each batch = 1 Claude API call with the full schema context + that area's criteria
- After all 9 area batches, fire 1 final call for the Executive Summary generation
- Total: ~10 API calls per audit

### Screen 3: Audit Report
**Route:** `/report`

Interactive report with tab navigation + XLSX download.

**Tabs:**
1. **Executive Summary** (new — not in current template)
2. **Audit Results** (the 45-row scored audit)
3. **Remediation Plan** (priority fixes extracted from results)
4. **Scoring Overview** (area-by-area breakdown with visual gauges)

## Tab Designs

### Tab 1: Executive Summary

**Content (generated by final LLM call):**

**Header block:**
- Customer name, date, project ID
- Overall grade badge (Needs Improvement / Meets Expectations / Exceeds Expectations)
- Overall score as fraction and percentage (e.g., 62/107 = 58%)

**Top Takeaways** — 3–5 bullet synthesis of the most important findings across all areas. Written for a VP-level reader who won't read the full audit.

**Short-Term Fixes (3)** — highest impact, lowest effort:
Each fix includes:
- What the issue is (1–2 sentences)
- Why it matters (business impact)
- How to fix it (concrete steps)
- Amplitude feature that helps (product-level rec + ACT-ID footnote)
  - Example: *"Use Amplitude Data's display name editor to normalize naming conventions without re-instrumentation."* ^ACT-004: rename_event_display^

**Medium-Term Fixes (2):**
Same structure but higher LOE. Typically requires engineering involvement.

**Long-Term Fixes (2):**
Same structure. Governance and process changes.

**For every gap/issue identified**, the summary maps it to one or more Amplitude capabilities:
- Amplitude Govern → schema enforcement, event blocking, status management
- Amplitude Data (Tracking Plan) → branching, schema validation, import
- Amplitude Data Transformations → property renaming, reclassification
- Amplitude Experiment → exposure tracking, A/B test properties
- Amplitude Groups → B2B account-level analysis
- Amplitude APIs → bulk remediation via Taxonomy API

### Tab 2: Audit Results

Interactive table matching the 14-column Output Template:

| Column | Editable? | Notes |
|--------|-----------|-------|
| Criteria ID | No | NS-001, SC-001, etc. |
| Area | No | Color-coded by area |
| Impact | No | Badge: High (red), Medium (amber), Low (green) |
| Criteria Topic | No | |
| Audit Score | No | Pass / Fail / N/A pill |
| Audit Comments | No | Expandable — truncated to 2 lines by default |
| Remediation Steps | No | Expandable |
| Amplitude Action | No | Links to ACT-IDs |
| Branch Name | Yes | Text input for user to fill post-audit |
| Change Status | Yes | Dropdown: Pending / In Progress / Complete / Skipped |
| Customer Sign-Off | Yes | Checkbox |
| Points Earned | No | Auto-calculated |
| Points Possible | No | From criteria definition |
| Priority Rank | No | High-impact fails ranked first |

**Filtering:** By area, by impact, by score (Pass/Fail/N/A), by change status.
**Sorting:** By priority rank (default), by area, by impact.

### Tab 3: Remediation Plan

Extracted from Fail rows, sorted by impact × effort matrix:

- **Immediate** (High impact, Low LOE) — things you can do in Amplitude Govern today
- **Short-term** (High impact, Medium LOE) — requires coordination but no engineering
- **Medium-term** (Medium+ impact, Medium LOE) — engineering involvement needed
- **Long-term** (Any impact, High LOE) — governance/process/cultural changes

Each item shows: criteria ID, issue summary, remediation steps, Amplitude feature callout (product rec + ACT-ID footnote).

### Tab 4: Scoring Overview

Visual dashboard:
- Radar/spider chart of 9 area scores (percentage per area)
- Area cards showing: X/Y points, pass rate, top fail
- Overall score bar with grade threshold markers

## LLM Prompt Architecture

### System Prompt (shared across all calls)

```
You are an Amplitude taxonomy audit agent. You evaluate Amplitude schema exports against a structured set of audit criteria and produce detailed, customer-facing findings.

## Rules
- Cite specific event and property names from the data — never use placeholders
- Write in informal, direct consultant tone using contractions
- Structure every Fail finding as: Finding → Evidence → Resolution
- Tailor all resolution steps to the customer's data source(s): {dataSources}
- Score: Pass = full points, Fail = 0, N/A = excluded from denominator
- Output valid JSON only — no markdown fences, no preamble

## Customer Context
- Customer: {customerName}
- Industry: {industry}
- Data Source(s): {dataSources}
- Compliance: {compliance}
- Known Concerns: {concerns}
```

### Per-Area Batch Prompt (9 calls)

```
Evaluate the following {areaName} criteria against the provided schema data.

## Schema Data
{parsedSchemaJSON}

## Criteria to Evaluate
{criteriaForThisArea}

For each criterion, return a JSON array:
[
  {
    "criteriaId": "NS-001",
    "score": "Pass" | "Fail" | "N/A",
    "comments": "...",
    "remediation": "..." or null,
    "amplitudeActions": ["ACT-004", "ACT-005"] or [],
    "pointsEarned": 1.0,
    "pointsPossible": 1.0,
    "priorityRank": null or 1-45
  }
]
```

### Executive Summary Prompt (1 call)

```
Based on the completed audit results below, generate an executive summary.

## Completed Audit Results
{allAuditResultsJSON}

## Amplitude Actions Catalogue
{actionsReference}

Return JSON:
{
  "overallScore": { "earned": N, "possible": N, "percentage": N, "grade": "..." },
  "topTakeaways": ["...", "...", "..."],
  "shortTermFixes": [
    {
      "title": "...",
      "issue": "...",
      "impact": "...",
      "steps": ["...", "..."],
      "amplitudeFeature": "...",
      "actIds": ["ACT-004"]
    }
  ],
  "mediumTermFixes": [...],
  "longTermFixes": [...]
}
```

## Key Implementation Notes

### CSV Parsing (csvParser.ts)
- Amplitude's Events CSV has these expected columns: `Event Name`, `Display Name`, `Description`, `Category`, `Status`, `Volume`, `Query Count`, `First Seen`, `Last Seen`
- Event Properties CSV nests under events — parse the parent-child relationship
- User Properties CSV is flat
- Validate column headers on upload; reject with clear error if schema doesn't match
- Strip BOM, normalize line endings, handle commas in quoted fields (PapaParse handles this)

### Claude API Calls (claude.ts)
- Model: `claude-sonnet-4-20250514`
- Max tokens: 4096 per call
- API key passed from form, stored in Zustand (session only)
- Implement retry with exponential backoff (3 attempts)
- Parse JSON response with fallback: strip markdown fences if present, then `JSON.parse`

### Pre-Processing Before LLM Calls
Do as much deterministic analysis client-side as possible to reduce token usage:
- Count events by status (LIVE, UNEXPECTED, BLOCKED, DELETED)
- Detect naming conventions (regex: Title Case, snake_case, camelCase, SCREAMING_SNAKE)
- Count properties per event
- Flag potential PII property names (email, phone, address, ssn, etc.)
- Calculate volume statistics (top 10 by volume, zero-query events)
- Detect duplicate property names across event/user scopes

Pass these pre-computed stats into the LLM prompt alongside raw schema data — this gives the model structured evidence to work from and reduces hallucination.

### Token Budget Estimate
- Typical Amplitude schema: 100–500 events × 5–20 properties = ~50K–200K tokens of schema data
- Strategy: For schemas >100K tokens, send only relevant subset per area batch (e.g., naming criteria only need event/property names, not full metadata)
- Executive summary call gets aggregated results (~5K tokens), not raw schema

## Cursor One-Shot Optimization Notes

When building the Cursor prompt from this plan:
1. **Inline all config data** — embed the 45 criteria, 20 actions, and scoring legend directly in the prompt rather than referencing external files
2. **Provide complete TypeScript interfaces** — define every type upfront so Cursor doesn't guess
3. **Include one complete component example** — e.g., fully implement IntakeForm so Cursor matches the pattern for remaining components
4. **Specify exact package versions** — avoid Cursor pulling incompatible versions
5. **Front-load the file structure** — Cursor works best when it knows the full tree before generating files
